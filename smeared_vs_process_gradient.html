How the gradient “smears” across the chain-of-thoughtRLOO outcome-RL

Why outcome reward touches every token equally — and how that differs from process supervision. Grounded in newcomb_rl/rloo.py.

1 · A completion’s probability factorizes per token

A completion \(y=(y_1,\dots,y_T)\) — the whole CoT plus the final answer token — is generated autoregressively, so its log-probability is a sum of one term per token:

$$\log \pi_\theta(y\mid x)=\sum_{t=1}^{T}\log \pi_\theta\!\left(y_t\mid x,\,y_{

Each \(\log\pi_\theta(y_t\mid\cdot)\) is just the log-softmax over the vocabulary at position \(t\).

2 · One scalar reward → one scalar advantage per completion

The reward \(R_i\) is computed once for completion \(i\) (the Newcomb payoff for its resolved action). RLOO turns the \(K\) rewards in a prompt-group into an advantage by leave-one-out:

$$A_i \;=\; R_i-\frac{1}{K-1}\sum_{j\ne i}R_j$$

One number per completion — not per token.

3 · The gradient — and exactly where the “smear” is

Vanilla policy gradient (clipping set aside for a second):

$$\nabla_\theta J=\frac{1}{N}\sum_i A_i\,\nabla_\theta\log\pi_\theta(y_i\mid x_i)=\frac{1}{N}\sum_i\sum_{t=1}^{T_i} \;\underbrace{A_i}_{\text{same for all }t}\;\nabla_\theta\log\pi_\theta\!\left(y_{i,t}\mid x_i,y_{i,
This is the smear, stated precisely. Look at the inner sum over \(t\): the same scalar \(A_i\) multiplies the gradient of every token — reasoning tokens and the answer token, identically. The per-token coefficient is constant in \(t\). The reward isn’t allocated (split) across tokens; it’s broadcast (copied) — every token sees the full \(A_i\).

4 · Your questions, answered directly

  • Are you upweighting each token in the trace? Yes. If \(A_i>0\), the step raises \(\log\pi_\theta(y_{i,t}\mid\cdot)\) for all \(t\) — it makes the whole sequence (CoT + answer) more probable next time. If \(A_i<0\), it lowers them all.
  • Is a separate derivative computed per token? Yes — \(\nabla_\theta\log\pi_\theta(y_{i,t}\mid\cdot)\) is a distinct quantity per token (ordinary backprop through the sequence). What’s shared is the scalar \(A_i\) in front of each.
  • How is the reward allocated across tokens? It is not allocated — it’s broadcast. No credit assignment happens; each token gets the full \(A_i\).
  • How much do you upweight by? Token \(t\)’s contribution is \(\propto A_i\) (identical for all \(t\) in the completion) \(\times\) learning rate, divided by the total token count (it’s a mean over tokens).

5 · In this code (RLOO + PPO-clipped surrogate + KL)

rloo.py lines 279–286 — the highlighted line is the smear made literal:

# per-token logπ_θ(y_t) for the generated tokens   →  shape [mb, T-1]
new_lp  = chunk_logprobs(ids, mask) * gi
adv_tok = adv[sl, None] * gi          # ← THE SMEAR: scalar A_i broadcast to every token
ratio   = exp(new_lp - old_lp)        # per-token ratio r_t = π_θ(y_t)/π_θ_old(y_t)
surr    = min(ratio*adv_tok, clip(ratio, 1±ε)*adv_tok)   # per-token clipped surrogate
kl      = exp(d) - d - 1              # per-token KL-to-base penalty
loss    = -(surr.sum() - β*kl.sum()) / total   # sum over ALL tokens, ÷ total token count

adv is [mb] (one scalar per completion); adv[:, None] is [mb, 1], broadcast across the [mb, T-1] token grid → the same \(A_i\) sitting on every token position. The resulting per-token loss is

$$L_{i,t} \;=\; -\min\!\Big(r_{i,t}\,A_i,\;\operatorname{clip}(r_{i,t},\,1{\pm}\epsilon)\,A_i\Big)\;+\;\beta\,\mathrm{KL}_{i,t}$$

with \(A_i\) identical for all \(t\). Clipping and KL are per-token gates on step size / divergence — but the reward signal riding through is still the one constant \(A_i\).

6 · The precise contrast with process supervision

Write the advantage with a token index, \(A_{i,t}\). The entire “smeared vs. process-based” distinction is one question: is the advantage a function of \(t\), or constant in \(t\)?

Outcome RLOO (what we run)Process supervision
advantage\(A_{i,t}=A_i\) — constant in \(t\)\(A_{i,t}\) varies with \(t\)
creditthe whole trace moves up/down as a block, by whether the final action paid offeach reasoning step is scored on its own (e.g. a process reward model / verifier)
consequencecorrect-reasoning→correct-answer and fluky-reasoning→correct-answer get the same \(A_i\); it can’t tell which token was “the good part”a valid derivation step gets a different coefficient than a fluky one, independently of the final answer

So outcome-RL can only ever push the whole trace’s probability up or down by one number. It has no lever to reward the reasoning as reasoning — which is exactly why it nudges disposition but doesn’t install the conditional rule.
